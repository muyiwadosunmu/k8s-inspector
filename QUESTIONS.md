# Written Questions

Question 1 — Error Handling & Resilience

The service must tolerate transient Kubernetes API failures and slow responses. I use context.Context with deadlines for every API call so the request handler cannot hang forever; timeouts are conservative (short) for user-facing endpoints and can be tuned. When an API call fails or times out I return a clear HTTP error (e.g., 503 Service Unavailable with an explanatory JSON body) rather than panicking. For transient errors I employ retries with exponential backoff for non-requests that alter state (with bounded retry attempts) and surfacing metrics for retry/error rates. In production I would add informers to cache frequently-read resources (pods, namespaces, nodes) to reduce API pressure and lower latency. I would also add a circuit breaker so that if the API is repeatedly failing the service returns a fast failure and warns via metrics/alerts. Finally, health checks should reflect API connectivity: readiness should fail if cached state is stale beyond a threshold so Kubernetes can avoid routing traffic to degraded instances.

Question 2 — Security

The provided RBAC manifests request only `get,list,watch` on `pods`, `namespaces`, and `nodes` because the service only needs to read cluster state. I bind these permissions to a dedicated ServiceAccount used by the Deployment. For multi-tenant clusters I would scope the role to namespace-level Roles where possible, avoiding ClusterRole when the service does not need cluster-wide visibility. Additional hardening includes: enabling Pod Security Standards, restricting the ServiceAccount to only the namespaces it needs, using NetworkPolicies to limit egress to known endpoints, and enabling admission controls like PodSecurityPolicy/PSP replacement to restrict capabilities. I would also enable IAM or cloud-provider-specific workload identities to avoid long-lived credentials, and ensure image provenance by enforcing image signature verification and deploying immutable image tags.

Question 3 — Observability

I would add structured logs (JSON) at appropriate levels: INFO for successful operations and DEBUG for verbose request/response details. Errors include contextual fields (namespace, pod name, request ID). Metrics: HTTP request latency histogram, request counts by endpoint and status code, Kubernetes API call latency and error counts, and cached vs live lookup counts. Alerting rules would trigger on increased 5xx rate, rising API error rates, or degraded readiness probes. Trace spans (OpenTelemetry) would show causal chains for slow requests and external API calls; sampling can be tuned. Exporting these to Prometheus/Grafana and a traces backend (Jaeger/Tempo) gives both live dashboards and distributed tracing to detect degradation before failure. Synthetic smoke checks can validate `/healthz` and `/summary` at regular intervals.

Question 4 — Extensibility

To list Deployments and rollout status, I would add new handlers that use the `apps/v1` client to get Deployments and their conditions (e.g., `status.conditions` and `status.updatedReplicas`). For Slack notifications on CrashLoopBackOff I would implement a controller-like informer that watches Pod events in watched namespaces and enqueues alerts when a pod enters CrashLoopBackOff. Key design choices: prefer informers and shared caches (from client-go) to avoid polling; decouple event detection from notification delivery by using a buffered worker queue and retryable webhook client; make the Slack notifier pluggable via an interface so different notifiers (Slack, PagerDuty) can be swapped. Security and rate-limiting are important: ensure notifiers don't leak secrets (store webhook tokens in Secrets) and throttle notifications to avoid spam during flapping. Tests should include unit tests for notification logic and integration tests using test fixtures.
